01The core idea

Two different questions we keep confusing

When an AI answers a cybersecurity question using retrieved documents, we can ask two separate things about its answer. Most research quietly treats them as one. This paper's whole point is that they are not.

Grounding

“Does the answer match the documents it was shown?” Also called faithfulness or attribution. It measures whether the answer looks sourced. It is cheap for a model to score high here: just paraphrase the text you were handed.

Factual correctness

“Is the answer actually true?” Measured against a human-checked reference answer the model never sees. This is what a security analyst actually needs. It's much harder — the model has to select the right facts, not just echo nearby text.

Why the difference is dangerous

A correct-but-unsourced answer is just hearsay. But an answer that is fluently grounded yet factually wrong is worse than useless — the citations make it persuasive. In cybersecurity that can mean acting on a fake vulnerability or mis-classifying a real attack. The paper calls this the “grounded-but-wrong” gap.

The finding in one line The design change that reliably makes answers look more sourced (GraphRAG) does not make them more correct — and sometimes makes them less correct. Only one method improved both.
02Background you'll need

How these systems actually work

All four systems tested are variations on RAG — Retrieval-Augmented Generation. The recipe: find relevant text, then let the model answer using it.

The vocabulary, once:
  • Chunk — a small slice of a document. The 49 PDFs became ≈8,250 chunks.
  • Embedding — the meaning of a chunk turned into a list of numbers, so the computer can compare meanings.
  • Cosine similarity — the score for “how close in meaning are these two texts?”
  • k — how many top chunks you feed the model (they tried 1, 3, 5).
  • Hallucination — a claim not backed by the given text (= 1 − grounding).

The four pipelines, side by side

A “pipeline” is a whole strategy for turning a question into an answer. Click each one — the diagram shows what changes. Crucially, the retrieval step is identical for the RAG ones, so any difference in results comes from what happens after retrieval.

GraphRAG adds one step: before answering, it follows a pre-built chunk graph to pull in extra “neighbour” chunks. DSPy instead swaps the hand-written prompt for a compiled, structured template (context + question → answer). Remember these two — they are the heroes and villains of the results.

03The test set

SENTINEL-QA: the benchmark they built

You can't measure any of this without a trustworthy set of questions with known right answers. So they built one from scratch — bilingual, and deliberately hard.

GPT-4o drafted question–answer pairs from single chunks of 49 authoritative cybersecurity PDFs (strategy documents, online-safety guides, laws). Then the humans took over: every question and reference answer was reviewed, edited, or thrown out. So it's a human-authored set, merely seeded by an LLM.

Two ground-truth labels per item: a reference answer (anchors correctness) and the set of reference chunk IDs (anchors grounding & retrieval scoring). English and Macedonian are both included on purpose.

Why only 80 items?Each item is answered by 4 pipelines × 3 models × 3 k-values, and every answer is scored by 4 judges — that's 16 judgments per (item, pipeline) cell. Scaling to thousands would be enormously expensive. 80 is a deliberate trade for depth over breadth. (Hold that thought — it's Limitation #2.)
What's in the benchmark
Difficulty is skewed toward hard on purpose — that's where grounding-only scoring is most misleading.

The ceiling nobody can beat: retrieval

Before comparing pipelines, note the hard limit. If the right chunk is never fetched, no amount of clever prompting can save the answer. Hit@5 tops out around 0.57–0.61. So on ~40% of questions, the correct source isn't even in the top 5. Every result below lives under this cap.

04Measurement

How 3,509 answers got graded

There's no human grading every answer — instead a panel of four AI models acts as judges (“LLM-as-judge”). Using four different models reduces any single model's bias.

The panel

GPT-4o, GPT-4o-mini, Claude Sonnet 4, DeepSeek-Chat. Three providers, two capability tiers. All run at temperature 0 for consistency.

Claim-by-claim

Each answer is split into sentences; every sentence is checked separately (the FactScore idea). Score = fraction of claims that pass.

Separated on purpose

The correctness judge sees the reference answer but not the retrieved context. The grounding judge sees the context but not the reference. Neither can fake one using the other.

Why this matters for the whole paperBecause the two judges look at different evidence, “grounding” and “correctness” are genuinely different measurements — not two names for the same score. That separation is what makes the decoupling finding meaningful.
05The evidence

Five ways of seeing the same result

The claim — grounding and correctness decouple; only DSPy lifts both — is shown five independent ways. If one could be a fluke, all five agreeing can't.

Evidence 1 · The grounding × correctness map
Each dot = one (pipeline, generator, k) setup. Hover any dot. Notice the shape: a wide horizontal spread, almost no vertical climb.
Read it like this: moving right = more grounded. Moving up = more correct. The RAG dots form a flat band around 0.20 correctness while grounding ranges from ~0.35 to ~0.82. DSPy (violet) is the only pipeline that reaches the upper-right — grounded and correct. GraphRAG (green) just shifts right, not up.
Evidence 2 · Pipeline scorecard (k = 3, averaged over generators)
Going down the list, grounding jumps a lot; correctness barely moves — until DSPy.
Grounding Factual correctness
From Classic → GraphRAG, grounding rises +0.12 but correctness only +0.03. From Classic → DSPy, correctness jumps +0.10 and grounding +0.30. The gap between the blue and orange bars is the grounded-but-wrong gap.
Evidence 3 · The GraphRAG “illusion”
Same questions, same retriever — only Classic → GraphRAG changes. We measure the paired change per item, then bootstrap it.
Δ Grounding Δ Factual
Bars are the average change when you switch to GraphRAG; whiskers are the 95% confidence interval. Grounding jumps and clears zero (real). Correctness hugs zero — and at k=5 on GPT-4o it goes negative (−0.048, significant). Grounding up, correctness flat-to-down.

Why “paired” is the trick

Comparing two averages can lie: maybe GraphRAG just got easier questions. Instead they compare each identical question answered both ways, and look at the difference. That rules out “luck of the draw.” The result: the very same items that gain grounding do not gain correctness. Since retrieval is shared, the split must come from the generation step, not from fetching better documents.

Plain-language bootstrap
A bootstrap is “reshuffle the data 10,000 times and see how often you'd get this result by chance.” If almost never (p < 0.05, the *), the effect is trusted.
Evidence 4 · Even the judges agree more about grounding than about truth
How much do the four judges agree with each other? Each row is a pair of judges. Farther right = more agreement (Spearman ρ).
Agreement on grounding Agreement on correctness
The blue dots sit right of the orange ones almost every time: judges agree on what looks sourced (ρ ≈ 0.64–0.85) more than on what is true (ρ ≈ 0.51–0.78). This is independent corroboration — truth is genuinely a harder, fuzzier target. GPT-4o-mini is the lenient outlier on correctness.
Evidence 5 · Cost vs correctness — the deployment case
Cheaper is left, more correct is up. The best options form the top-left frontier. Hover any point.
Only DSPy points sit on the frontier. DSPy at k=3 costs ~$0.0024/call and runs in 0.97s; Classic RAG at the same k costs ~$0.0044 and takes 2.25s. So DSPy is ~1.8× cheaper, ~2.3× faster, and more correct. Every GraphRAG point is beaten by some DSPy point.
So what closes the gap?Not fetching more text (GraphRAG). It's structuring the prompt (DSPy). The authors' hypothesis: DSPy's rigid context → question → answer schema makes the model treat retrieved chunks as a reference library to pick from, rather than a story to keep writing — so it selects claims that are actually backed, instead of just plausible-sounding continuations.

06Stress tests

Does the pattern hold when you poke it?

A finding is only trustworthy if it survives being sliced different ways — by k, by language, by difficulty — and if you can explain why individual answers win or lose.

More context (k) lifts grounding, not correctness
GPT-4o. Solid = correctness, dashed = grounding, as k goes 1 → 3 → 5.
Factual (solid) Grounding (dashed)
Grounding climbs with k for every pipeline. Correctness doesn't: GraphRAG even makes an inverted-U, dropping at k=5. DSPy at k=1 already beats Classic at k=5 on correctness — “compile the prompt” beats “retrieve 5× more.”
Split by difficulty — everything collapses on hard items
Factual correctness only. Watch the hard bars.
On hard questions every pipeline falls to ~0.02–0.08 correctness, even DSPy — yet grounding stays high. That's the danger zone: confident, well-cited, wrong. The gap is widest exactly where it's most harmful.
Language & generators, briefly
Language: grounding is higher in Macedonian (the model reuses the Macedonian source text) but correctness is flat or lower — so the split is if anything sharper in the low-resource language, not an English quirk. Generators: GPT-4o and Claude (49 paired items each) carry the headline; DeepSeek (only 4 items) is a small sanity check — same direction, too little data to be sure.

The four failure modes (why answers win or lose)

They hand-examined the 10 items where GraphRAG and Classic disagree most, and sorted the reasons into four modes:

M1Synthesis benefit — GraphRAG wins

The answer needs several adjacent passages stitched together (e.g. the full list of a DMZ's roles). Graph expansion supplies the missing neighbours. This is GraphRAG working as intended.

M2Context dilution — Classic wins

A simple lookup that one good chunk already answers. Graph expansion drags in extra text and distracts the model away from the exact fact.

M3Enumeration dilution

The right answer is a fixed list (laws, steps). Extra context tempts the model to pad the list with invented items — worse in Macedonian, where its priors are weaker.

M4Grounded-but-wrong — both fail

Both pipelines produce fluent, well-grounded answers (grounding ≥ 0.5) that are still mostly false (correctness < 0.2). A grounding-only evaluation would call these a success. This is the whole paper's warning, made concrete.

07For your professor

Limitations — and how to defend them

This is the section examiners push on. The goal isn't to pretend the study is perfect — it's to show you understand each limit, why the authors made that choice, and what it does (and doesn't) threaten. Each card below: what it means, why it's OK, and a likely question with an answer.

core threat could undermine the main claim scope limits how far it generalises contained acknowledged, low risk
L1 The judges are AI, not humans core threatEvery score comes from LLM judges, never validated against a human-annotated subset. ▶
What it means
The two headline metrics (grounding, correctness) are produced by four language models. No human went back and confirmed a sample of those scores, so in principle the whole ranking could reflect what models think is correct, not what is correct.
Why the authors say it's acceptable
They call it a deliberate scope choice, not a forgotten step. Their defence has three parts: (1) they use a diverse 4-model panel across 3 providers, so no single model's bias dominates; (2) they report inter-judge agreement for every number instead of hiding it — and that agreement gap is itself a finding; (3) at 3,509 judgments, a multi-judge panel is “the minimum defensible substitute for human anchoring” at that scale.
Why it doesn't sink the main claim
The central result is a relative one: grounding rises while correctness doesn't, on the same judges. Even a biased judge, applied equally to both pipelines, would still reveal the decoupling. The bias would have to be different for grounding vs correctness in a very specific way to fake this — and the judge-agreement split is independent evidence that it isn't.
Q: “Aren't you just measuring what GPT-4o believes, not the truth?”
A: For absolute scores, partly yes — that's why we never claim an absolute accuracy number is authoritative. But the finding is a paired, relative comparison under identical judges, so a shared judge bias cancels out. And the fix is already named as future work #1: a human-annotated calibration subset.
L2 Only 80 questions scopeSmall benchmark; some cells (especially hard items, DeepSeek) have little data. ▶
What it means
80 items is small next to benchmarks with thousands. Statistical power on sub-slices (hard-only, Macedonian-only) is limited, and DeepSeek only has 4 paired items — too few to conclude anything on its own.
Why the authors say it's acceptable
The size buys depth: 16 judgments per (item, pipeline) cell across the full grid. Bigger benchmarks trade that multi-judge depth for scale. The main effects (grounding lift, DSPy dominance) still reach significance at n = 49 paired items on the two headline models, via the paired bootstrap.
Honest residual risk
The hard-item conclusions are the weakest — that's exactly where correctness collapses and where more data is most wanted. The authors flag “a larger benchmark tilted toward hard items” as future work, and treat DeepSeek explicitly as a qualitative probe, never a headline number.
Q: “Is 80 enough to trust?”
A: For the headline paired comparisons on GPT-4o and Claude, yes — the bootstrap confidence intervals exclude zero. For fine-grained slices and DeepSeek, no, and we say so rather than over-claiming. Scaling is “budget-linear,” i.e. just expensive, not hard.
L3 DSPy vs Classic changes two things at once core threatDSPy is both structured and terser — so which one causes the win? ▶
What it means
Compared to Classic RAG, the DSPy prompt differs in two ways: it's structured (an explicit context/question/answer schema) and it drops the long “cybersecurity analyst” persona wrapper (it's shorter). The experiment has no “Classic-but-short” control, so we can't cleanly say whether structure or brevity carries the improvement.
Why the authors say it's acceptable
They are careful with the wording: they claim the bundle (structure + minimal prompt) closes the gap at their operating point — they do not claim to have isolated the mechanism. This is the intellectually honest move: don't over-attribute.
Why it's a real limitation, not a fatal one
It doesn't threaten the decoupling finding (that's about GraphRAG, not DSPy). It only limits how precisely we can explain why DSPy wins. Future work #5 is the exact fix: a 4-cell control {Classic-long, Classic-minimal, DSPy-long, DSPy-minimal} to separate the two effects.
Q: “Maybe DSPy just wins because its prompt is shorter?”
A: Possibly — that's the honest answer, and it's why we frame it as “structure + brevity as a bundle.” The paper's stronger, separable claim is the negative one about GraphRAG; the DSPy result is a strong positive lead that needs the ablation to fully explain.
L4 DSPy only ran on GPT-4o scopeThe star pipeline was tested with one model, for tooling reasons. ▶
What it means
DSPy — the pipeline that “wins” — was only run with GPT-4o, because the DSPy runtime at the time needed a specific openai/<model> configuration and stable adapter support. So we can't be sure the DSPy advantage transfers to Claude or DeepSeek.
Why it's contained
The paper's main claim (grounding ≠ correctness under GraphRAG) is shown on both GPT-4o and Claude, so it can't be a DSPy-or-OpenAI artifact. DSPy is presented as “the resolution on this benchmark,” explicitly not “the globally best pipeline.”
Q: “How do you know DSPy isn't just GPT-4o being good?”
A: We don't fully — that's the limit. But the decoupling that DSPy resolves is visible across two model families, so the phenomenon is real independent of DSPy. Cross-model DSPy replication is named future work #4.
L5 The mechanism is a hypothesis, not proof contained“Why” DSPy helps is an argued conjecture, not a demonstrated cause. ▶
What it means
The explanation — that structure makes the model “select from a reference library rather than continue a story” — is a plausible hypothesis consistent with the data, but not directly demonstrated. Proving it needs looking inside the model (attention patterns, attribution patching).
Why it's acceptable
The empirical pattern stands on its own regardless of the mechanism. The authors clearly label it a hypothesis and list the instrumentation needed to upgrade it (future work #6). Good science: separate what happened (solid) from why (proposed).
Q: “Can you prove that's why DSPy works?”
A: Not yet — it's our best explanation and it fits every slice of the data, but confirming it requires mechanistic (inside-the-model) analysis, which we scope as future work.
L6 Narrow domain, two languages, one retriever, no graph sweep scopeCybersecurity only; EN+MK only; fixed embedder; GraphRAG's knobs weren't fully explored. ▶
What it means
Four smaller boundaries bundled: (a) one domain (cybersecurity); (b) two languages, not broad multilingual; (c) a single embedding model (MiniLM-L12-v2); (d) GraphRAG's τ (similarity threshold) and expansion budget weren't swept — one setting (τ=0.8, 10 nodes) was used for the headline.
Why the authors say it's acceptable
They argue the mechanism should appear wherever three conditions hold: a redundant knowledge base (legal, medical, technical manuals all qualify), a mix of synthesis + lookup questions, and a capable-enough generator. Where those hold, expect the same split. Where they don't (e.g. one gold passage per question), it should fade. The graph sweep is anchored by paired Classic-vs-Graph deltas at a fixed setting, so it doesn't depend on the un-run sweep.
Q: “Would this replicate outside cybersecurity / with a bigger graph?”
A: We predict yes for any redundant, high-stakes corpus, and give the falsifiable conditions. The full graph-topology heatmap is future work #2 — but our claim rests on matched-pair deltas, not on having found GraphRAG's best settings.
The 30-second version for the examThe one limitation that could actually wound the study is L1 (AI judges) — and the defence is that the finding is a paired, relative comparison under identical judges, plus the judge-agreement split is independent evidence. Everything else (80 items, DSPy-only-GPT-4o, structure-vs-brevity, mechanism-as-hypothesis) limits scope or precision, not the core decoupling claim — and each has a named future-work fix.
08The repository

How the code produces the paper

The repo is two things: a working cybersecurity Q&A app (FastAPI + Gradio), and a research pipeline that runs the whole experiment end-to-end. Every figure in the paper is reproducible by running these steps in order.

1
Build the search index
rag/create_database.py chops the 49 PDFs into ≈8,250 chunks and embeds them into a Chroma vector database. rag/graph/build_graph.py then builds the chunk graphs (three similarity thresholds for the ablation).
→ the shared retrieval substrate everything else uses
2
Create the benchmark
bench/generate_questions.py drafts Q&A from chunks; bench/populate_reference_doc_ids.py maps each item to its reference chunks. Humans then edit the result into data/benchmark.jsonl (80 items).
→ SENTINEL-QA (§3, Table 1)
3
Generate answers
bench/run_benchmark.py runs the four pipelines in rag/pipelines.py across 3 generators × k ∈ {1,3,5}. Mode strings like rag_graph:openai:k=5:mn=10 encode the full config. Fully resumable.
→ raw answers for all 21 RAG configs (§4)
4
Judge every answer
bench/meta_eval.py runs the 4-judge panel using judge_prompt_ground.py and judge_prompt_factual.py, scoring grounding + correctness claim-by-claim. 3,509 judgments; grounding skipped for zero-shot.
→ the evaluation grid (§4)
5
Analyse & test
analyze_judgements.py aggregates; metrics.py computes P@k / Hit@k, latency & cost; bootstrap_tests.py runs the 10,000-resample paired tests; error_analysis.py builds the M1–M4 case studies.
→ Tables 2–5, the numbers behind every claim
6
Draw the figures
bench/figures.py and bench/tables.py emit every PDF figure and LaTeX table straight into the paper folder.
→ Figures 1–6 + supplements
Supporting modules
  • rag/config.py — all keys, model names, paths
  • rag/llm_factory.py — one factory for OpenAI / Claude / Gemini / DeepSeek
  • rag/retrieval.py — the shared top-k vector search
  • rag/graph/expand.py — the graph neighbour-expansion step
  • rag/prompts.py — the shared multilingual prompt template
The app side (the “product”)
  • main.py / backend.py — FastAPI service
  • ui_gradio.py — the research chat UI
  • Dockerfile — containerised deploy

The research pipeline (bench/) and the app (main.py) share the same rag/ core — so what you benchmark is what you'd ship.

09One-page recall

Exam cheat-sheet

The claim, three sentences

In cybersecurity RAG, grounding and factual correctness decouple. GraphRAG reliably raises grounding (+0.09 to +0.14) but leaves correctness flat — and at k=5 lowers it (−0.05). DSPy (structured prompting) is the only pipeline that raises both, while being cheaper and faster.

The takeaway

Grounding alone is not a safe proxy for correctness. Report both, with judge-agreement and paired statistics. To improve faithfulness, compile the prompt, don't just expand retrieval.
